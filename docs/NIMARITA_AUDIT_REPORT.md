# Аудит репозитория Nimarita

## Статус аудита

- Источник аудита: архив `nimarita_final_rebuilt.zip`
- Формат аудита: статический разбор кода + запуск тестов + проверка импортируемости/компилируемости
- Проверки, которые были выполнены:
  - `python -m pytest -q` → `21 passed`
  - `python -m compileall -q .` → `OK`
- Важно: в архиве **нет git-истории**, поэтому changelog ниже является **реконструкцией по текущему состоянию кода**, а не дословной историей коммитов.

## Executive summary

`Nimarita` — это Telegram-бот и Telegram Mini App для пары **1↔1**. Продукт позволяет:

- создать и подтвердить пару;
- выбрать роли в паре (`девушка` / `парень` / `не указано`);
- ставить напоминания партнёру;
- использовать готовые заботливые сообщения;
- отправлять свои сообщения и отвечать на сообщения партнёра;
- управлять всем как из Telegram-бота, так и через Mini App.

Архитектурно проект собран аккуратно: есть слой домена, отдельные репозитории, сервисы, веб-слой, телеграм-слой и фоновые воркеры. Главная бизнес-инварианта — **одна активная пара на пользователя** — защищена и на уровне сервисов, и на уровне SQLite-триггеров.

В текущем состоянии проект выглядит как **рабочий single-instance продукт на SQLite**, хорошо подходящий для деплоя на Railway **с подключённым Volume и одной репликой**.

## Что реально происходит в приложении

### Продуктовая модель

1. Пользователь делает `/start` в боте.
2. Пользователь получает основную клавиатуру с кнопкой входа в Mini App.
3. Один из пользователей создаёт invite в пару.
4. Второй пользователь принимает invite.
5. После активации пары становятся доступны:
   - reminders;
   - care template catalog;
   - custom care;
   - ответы на care-сообщения;
   - Mini App-дэшборд.

### Два интерфейса поверх одного домена

В проекте фактически два UX-слоя поверх одной и той же бизнес-логики:

- **Telegram bot UX** (`nimarita/telegram/*`)
- **Mini App UX** (`nimarita/web/server.py` + `nimarita/web/static/index.html`)

Оба слоя работают через одни и те же `services/*`, поэтому поведение достаточно консистентное.

### Как доставляются reminders и care

Доставка сделана по схеме **outbox + background workers**:

- reminder создаёт правило + occurrence в БД;
- care создаёт dispatch в БД;
- воркеры периодически забирают due-задачи;
- Telegram notifier доставляет сообщение;
- после успеха запись помечается delivered/sent;
- при ошибке выполняется retry/backoff;
- при финальной неудаче отправляется уведомление инициатору.

Это хороший паттерн для надёжности: пользовательское действие не зависит от синхронной отправки сообщения в тот же момент.

## Сильные стороны репозитория

### 1. Чёткое разделение ответственности

Проект не смешивает всё в одном месте:

- domain хранит модели/enum/ошибки;
- repositories работают с хранилищем;
- services держат бизнес-правила;
- telegram/web — только интерфейсные слои;
- workers — фоновая доставка и обслуживание.

### 2. Сильная защита инварианты 1↔1

В коде явно реализовано правило «один пользователь не может иметь несколько активных пар». Это защищено:

- логикой `PairingService`;
- ограничениями и индексами БД;
- SQLite triggers `trg_pairs_prevent_multi_active_insert/update`.

### 3. Хорошая operational hygiene для SQLite

Есть всё, чего обычно не хватает маленьким проектам на SQLite:

- startup quick-check;
- foreign key check;
- scheduled checkpoint;
- scheduled backups;
- graceful shutdown checkpoint;
- stale-processing recovery;
- readiness payload с heartbeat воркеров.

### 4. Корректная модель Telegram WebApp auth

`initData` не доверяется фронтенду: сервер сам валидирует подпись и срок годности. Это правильная модель для Mini App.

### 5. Неплохое покрытие тестами по критичным сценариям

Тесты есть по:

- pairing;
- access policy;
- reliability/retry;
- reminder recurrence;
- care catalog и кастомным ответам;
- SQLite hardening.

## Риски и технический долг

### 1. Отсутствует настоящая git-история в архиве

Для нового разработчика это значит:

- невозможно посмотреть эволюцию решения по коммитам;
- сложно понять, что было legacy, а что текущий контракт;
- changelog приходится реконструировать вручную.

### 2. Frontend монолитный

`nimarita/web/static/index.html` — это сразу:

- HTML;
- весь CSS;
- весь JS;
- state rendering;
- tabs;
- forms;
- notices;
- fetch client.

Это удобно для простого деплоя, но тяжело в поддержке. Любое серьёзное изменение веба будет концентрироваться в одном большом файле.

### 3. Веб-сессия stateless и без revoke

`SessionManager` использует HMAC-подписанный токен без server-side session store. Это нормально для малого продукта, но нужно понимать ограничения:

- нельзя адресно отозвать одну сессию;
- ротация секрета инвалидирует все сессии сразу;
- нет списка активных веб-сессий.

### 4. Нет отдельной migration-системы

Совместимость БД сделана через additive-миграции внутри `nimarita/infra/sqlite.py`. Для текущего масштаба это рабочая модель, но при дальнейшем росте лучше перейти на явный migration registry.

### 5. SQLite = только single-instance

Сам код и конфиг подразумевают single-instance runtime. При нескольких репликах можно получить гонки доставки, конкуренцию за БД и проблемы консистентности. Для Railway это особенно важно.

### 6. Веб-защита умеренная, но не максимальная

Есть:

- `Cache-Control: no-store`
- `X-Content-Type-Options: nosniff`
- `Referrer-Policy: no-referrer`
- CORS whitelist

Но отсутствуют, например:

- CSP;
- HSTS на уровне приложения;
- явная защита от iframe embedding на уровне заголовков.

Часть из этого можно закрыть reverse proxy / платформой, но в коде приложения этого сейчас нет.

## Что уже хорошо сделано по продукту

### Роли пары

Введены роли `woman / man / unspecified`, и care-шаблоны фильтруются по sender/recipient role.

### Gender-aware care

В каталоге реализована идея, что часть шаблонов — строго role-specific. Это уже превращает care-раздел из абстрактного «слоя сообщений» в более осмысленный продуктовый контент.

### Recurring reminders

В системе поддерживаются:

- one-time;
- daily;
- weekdays;
- weekly;
- interval (`hour/day/week/month`, `N >= 1`).

### Web UX стал значительно богаче базового shell

В Mini App уже есть:

- tabs `Пара / Напоминания / Сообщения`;
- profile card;
- invite flow;
- grouped reminders;
- edit/cancel reminder;
- care template list;
- care history;
- custom care и custom replies;
- notifications/notices;
- периодический refresh state.

## Итоговая оценка

### Текущее состояние

**Проект в рабочем состоянии.** Это уже не «черновик» и не набор разрозненных файлов, а цельный runtime.

### К чему он готов

- закрытый beta-rollout;
- single-instance production на Railway;
- повседневное использование ограниченным количеством пользователей;
- дальнейшая итерация по UX и контенту.

### Что я бы делал следующим этапом

#### P0 — обязательно

1. Зафиксировать документацию (сделано текущим пакетом).
2. Держать SQLite только на Railway Volume.
3. Явно держать **1 replica**.
4. Регулярно проверять `/health/ready`.

#### P1 — желательно

1. Разбить frontend на модули хотя бы логически, если не физически.
2. Добавить snapshot/integration tests для Mini App UI.
3. Ввести более формальный migration ledger.

#### P2 — по мере роста

1. Добавить server-side rate limiting для web API.
2. Добавить CSP и усилить security headers.
3. При масштабировании вынести БД из SQLite в managed Postgres.

## Краткий вывод для разработчика

Если открыть репозиторий впервые, правильная ментальная модель такая:

> Это Telegram-first продукт для пары 1↔1, где бот и Mini App — просто два интерфейса над единым доменом. Бизнес-правила живут в `services`, хранение — в SQLite через `repositories`, а доставка reminders/care сделана воркерами через outbox-подобную схему.
